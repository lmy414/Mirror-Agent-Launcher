# Core-Extension Philosophy（核心扩展分离）

## 核心 vs 扩展

每个项目拆解为**核心**和**扩展**两部分：

| | 核心 | 扩展 |
|------|------|------|
| 角色 | 项目地基，不可变基线 | 可增删改的功能模块 |
| 稳定性 | 设计冻结后不进行大范围改动 | 随时可迭代 |
| 依赖 | 不依赖任何扩展 | 只能依赖核心，不能依赖其他扩展 |
| 通信 | 定义接口和消息规范 | 通过核心提供的统一机制收发消息/事件 |
| 示例 | ConfigAdapter 接口、Registry、IPC 通道 | ClaudeCodeAdapter、Sidebar、TerminalView |

**规则**：
- 扩展之间零依赖。两个扩展有相同逻辑时 → 抽离到核心工具库
- 新增能力 = 扩展核心接口（加方法），不修改已有接口签名
- **判断放哪里**：删除这个功能，核心还能不能跑？能 → 扩展；不能 → 核心

## 单文件约束

| 规则 | 理由 |
|------|------|
| 一个文件只做一件事 | 降低理解成本，AI 和人都能一眼看懂 |
| 单文件不超过 300 行 | 认知负载上限，超过强制拆分 |
| 拆分方式 | `Component.tsx` → `Component/SubFeature.tsx` + `utils.ts` |

## 共享逻辑提取

当两个扩展出现相同的逻辑或函数时：

```
扩展 A ──┐
          ├── 相同逻辑 → 提取到核心工具库（main/utils/ 或 src/shell/）
扩展 B ──┘
```

提取后两个扩展都从核心引用，不再各自维护副本。

## 审查清单

每次修改代码时检查：

- [ ] 新增代码放对了核心/扩展位置？
- [ ] 扩展之间是否存在直接 import（跨扩展引用）？
- [ ] 新增文件是否超过 300 行？
- [ ] 两个扩展是否出现了重复逻辑 → 应提取？
- [ ] 修改核心接口时，是否只新增方法而不改已有签名？
