# Regression Debugging（回归调试）

## 核心原则

**先理解，再动手。让 Bug 自己说话，不要猜。**

---

## 七步法流程

### Step 1: STOP —— 暂停，收集信息

**禁止直接给 diff。** 先确认：

- Bug 的**最小复现路径**（精确到点击哪个按钮、看到什么）
- **最近变更**：过去 10 次 commit / 本次 session 中修改了哪些文件
- 是否有测试覆盖该路径

如果信息不足，先向用户提问补充。

### Step 2: DIFF —— 梳理变更，列出嫌疑清单

用 git diff 或回顾本次 session，列出所有改动文件。对每个文件标记**嫌疑等级**：

| 嫌疑 | 标记 | 说明 |
|------|------|------|
| 🔴 高 | 数据流链路文件 | 修改了接口签名、IPC 通道、数据传递路径 |
| 🟡 中 | 渲染/展示文件 | 修改了组件结构、条件渲染、样式布局 |
| 🟢 低 | 纯工具/配置 | 格式化、常量定义、不影响运行时的改动 |

**重点检查**：
- 文件迁移时旧编译产物是否残留（`.js` / `.js.map` 未清理）
- 接口签名变更后所有调用方是否同步更新
- 模块路径变更后 `require` / `import` 是否仍正确解析

### Step 3: TRACE —— 画出数据流，标出断点

从用户操作开始，逐层追踪数据经过的完整路径：

```
用户点击 → 组件事件 → IPC invoke → preload → 主进程 handler
                                                    ↓
         渲染更新 ← IPC on/stdout ← preload ← 返回/推送
```

对每个环节标注：
- ✅ 已验证通过的环节
- ❓ 未验证的环节（最可能的断点）

**IPC 数据流自检清单**（来自 electron-ipc-sync）：

| 检查项 | 方法 |
|--------|------|
| 通道名是否一致 | 主进程 `ipcMain.handle('x:y')` vs preload `ipcRenderer.invoke('x:y')` |
| preload 方法是否正确包装 | `invoke` 返回值是否被正确 `return` |
| sender 定向发送 | `event.sender.send()` 只发给创建者，不是所有窗口 |
| 事件监听是否已清理 | 组件 `onCleanup` 中是否调用了 unsubscribe |
| 主进程编译产物是否最新 | `dist-electron/` 下是否有多余的陈旧 `.js` 文件 |

### Step 4: VISUALIZE —— 加可视化诊断日志

当 DevTools 打不开或无法远程调试时，**把诊断信息渲染到页面上**：

```tsx
// 可复制的文本框——用户全选复制发给你
<textarea readonly style={{
  width: '100%', background: 'rgba(0,0,0,0.8)', color: '#0f0',
  'font-family': 'monospace', 'font-size': '10px', 'user-select': 'all',
}}
  value={`VAR_NAME: key1=val1 key2=val2`}
/>
```

**诊断信息必须覆盖数据流的每个关键节点**：

| 节点 | 诊断内容 |
|------|---------|
| 组件状态 | 数据数组长度、当前选中 ID、条件判断结果 |
| IPC 调用结果 | `ok` 状态、返回数据长度、错误信息 |
| 渲染条件 | 各 `Show` / `if` 条件的真假值 |
| 异步加载 | 数据是否到达、Promise 是否 resolve |

**原则**：一个 `textarea` 放一层的数据，分层标注。让用户复制粘贴即可，不需要截图。

### Step 5: ANALYZE —— 对比预期与实际

拿到用户的诊断输出后，与预期的数据流对比：

```markdown
## 数据流对比

| 环节 | 预期 | 实际 | 状态 |
|------|------|------|------|
| 组件挂载 | selectedId='claude-code' | selectedId='claude-code' | ✅ |
| IPC 调用 | ok:true, data:[...] | ok:true, data:[] | ❌ |
| 渲染 | hasProviders=true | hasProviders=false | ❌ |
```

断点位置 = 第一个 **预期 ≠ 实际** 的环节。

**常见根因模式**：

| 现象 | 常见根因 |
|------|---------|
| IPC 返回空数组 | 主进程编译产物陈旧；适配器方法缺失 |
| 数据流到但页面不显示 | 竞态条件：监听器注册晚于数据到达 |
| 点击无反应 | 信号未正确更新；`Show` 条件永远 false |
| 闪现后消失 | spawn 成功但立即 exit；环境变量不完整 |
| 模块路径变更后崩溃 | `require` 先匹配到旧文件（`.js` 优先于 `index.js`） |

### Step 6: FIX —— 最小安全修改

给出修复方案时，必须附带：
- 每个改动的**理由**（为什么改这里而不是别处）
- 不改哪些相关代码（明确边界）
- 是否需要同步更新编译产物、类型声明

**禁止事项**（除非经过明确论证）：
- 用 `any` / 类型断言绕过类型检查
- 删除或跳过现有测试
- 修改公共接口签名而不更新所有调用方
- 保留陈旧编译产物（`.js` / `.js.map` / `.d.ts`）

### Step 7: VERIFY —— 回归验证

修改完成后必须验证：
1. 复现步骤现在不再触发 Bug
2. 全部现有测试通过（`npm test`）
3. 类型检查通过（`tsc --noEmit`）
4. 如果涉及主进程或原生模块，必须重新编译并重启
5. 清理诊断日志（不应留在生产代码中）

---

## 预防清单

修复完成后，自检以下问题：

- [ ] 是否清理了所有诊断代码？
- [ ] 旧编译产物是否需要手动删除（路径变更/文件迁移时）？
- [ ] 新增的 IPC 通道是否在主进程、preload、类型声明三处同步？
- [ ] 是否新增了测试覆盖本次修复？
- [ ] 是否有竞态条件风险（数据到达早于监听器注册）？
- [ ] 事件监听器是否在组件卸载时正确清理？

## 用户交互模板

当出现反复 Bug 时，对 AI 使用：

```
这个 Bug [描述]。在给出任何修复代码之前，请：
1. 梳理最近变更，列出嫌疑文件清单
2. 画出完整数据流，标出已验证和未验证的环节
3. 在每个关键节点添加可复制的可视化诊断
4. 让我复制诊断输出，你再做深度分析
5. 给出修复方案并说明为什么这是根因而非表象
```

## AI 自检模板

收到用户 Bug 报告后，AI 必须自问：

1. 我最近是否修改了相关文件？列出所有改动。
2. 数据从 A 到 B 的完整路径是什么？每个环节的输入输出类型是否匹配？
3. 用户能打开 DevTools 吗？如果不能，我是否加了可复制的诊断文本？
4. 如果是路径变更，旧编译产物是否已清理？
5. 修复后我是否清理了所有临时诊断代码？
