# Incremental Iteration（增量迭代）

## 定位

与 superpowers 同级的工作流技能。用于需求模糊、需要快速出东西、UI 密集的场景。不做一次性的完整设计，而是分阶段收需求、拆解、逐项交付。

## 核心原则

1. **想到哪里做到哪里** — 上一步是下一步的基础，两个模块不耦合时可以并行
2. **最小可验收单元** — 每次交付用户可感知的变化
3. **UI 架构先行** — 组件库和样式体系在视觉需求之前就位
4. **按空间区域拆 UX** — 以大区块为父需求，交互反馈为子，细节为孙

---

## 工作流总览

```
收集 ──→ 审查 ──→ 拆解 ──→ 逐项执行 → 全部完成
  │        │        │           │
  │        │        │           ├── 冲突？→ 暂停 + 弹窗 + 建议
  │        │        │           └── 通过？→ 下一个
  │        │        │
  │        │        └── 子需求边界不清？→ 递归
  │        │
  │        └── 用户补充 → 回审查
  │
  └── 用户选择「给我点建议」→ AI 提示遗漏方向
```

---

## 第一阶段：收集（只收需求，不动手）

### AI 行为

1. 用户说出一段需求 → AI 记录到需求文档（不执行）
2. 引导用户继续说
3. **绝对禁止**：此阶段不写代码、不设计架构、不给出实现方案

### 交互控制（使用 AskUserQuestion 弹窗）

每轮用户输入后，AI 弹出三个选项：

| 选项 | 行为 |
|------|------|
| ✅ 确认，进入审查 | 结束收集，开始第二阶段 |
| 💬 继续补充 | 关闭弹窗，等待用户新一轮输入 |
| 💡 给我点建议 | AI 回顾已收集的需求，提示可能的遗漏方向（只提示维度，不补充具体内容） |

### 示例

```
用户: 我要做一个工作目录管理功能，设置页能添加/删除路径，新建终端时能选。

AI: [弹窗] 确认 / 继续补充 / 给我点建议

用户: [继续补充] 路径要持久化，还有每个 Agent 可以单独开关要不要每次询问。

AI: [弹窗] 确认 / 继续补充 / 给我点建议

用户: [确认，进入审查]
```

---

## 第二阶段：审查（AI 补漏，不写代码）

### AI 行为

按优先级逐层审查用户需求，每层扫描完后输出：

```
## P0 安全审查
- ✅ 无安全风险
- ⚠️ 发现：路径由用户手动输入，未校验防止路径遍历
  建议：添加路径白名单校验
- ❌ 缺失：API key 在配置中明文存储
  建议：使用 Electron safeStorage 加密

## P1 依赖审查
...
```

### 审查优先级

| 优先级 | 维度 | 如果漏了 | 检查点 |
|--------|------|---------|--------|
| **P0** | 安全 | 架构级返工 | 输入校验、路径遍历、RCE、API key 泄露、权限越界 |
| **P1** | 依赖关系 | 推倒重来 | 需求 3 是否依赖需求 1 的产出？做反了需重来 |
| **P2** | 生命周期 | 用不了 | 启动/运行/退出/恢复/错误处理/状态持久化 |
| **P3** | UI/UX 边界 | 体验差 | 空状态、加载态、错误态、边界数据、响应式 |
| **P4** | 代码规范 | 可后修 | 文件行数、命名一致性、样式抽离、类型覆盖 |

**原则**：假设用户 0 编码经验，发现问题用用户能理解的语言解释。不要跳过去假设用户知道。

### 审查后交互

输出审查结果 → 弹窗：「确认，开始拆解」/「根据建议补充需求」

---

## 第三阶段：拆解（保持用户顺序）

### 规则

1. **默认按用户说的顺序**拆解。用户说 1,2,3,4 → 执行顺序就是 1→2→3→4
2. **无法按顺序时**（需求 2 依赖需求 4 的产物）→ 告诉用户为什么调整，给出调整后顺序，等确认
3. **子需求递归**：
   - 边界清晰 → 直接拆成执行步骤
   - 边界模糊 → 询问用户不清晰的点 → 收集补充 → 拆解

### 拆解粒度

每个最小单元以**用户可感知的功能变化**为界：

```
用户需求: "加一个深色模式"
├── 步骤 1: CSS 变量体系 + 色值定义（不可见，基础设施）
├── 步骤 2: 添加切换按钮（用户可感知的 UI 变化）
└── 步骤 3: 持久化到设置（用户可感知：刷新后保持）
```

### 拆解后确认

输出完整拆解方案 → 弹窗：「确认，开始执行」/「调整」

---

## 第四阶段：逐项执行

### 规则

1. 执行步骤 1 → 测试 → 用户验收 → 通过后执行步骤 2
2. **每个步骤之间暂停**，等用户确认
3. 两个完全不耦合的模块 → 可以同时执行（声明依赖关系给用户确认）

### 冲突处理

执行步骤 3 时，发现和已完成的步骤 1 冲突：

1. **立即暂停**全部执行
2. 弹窗告知用户：
   - 冲突内容（步骤 3 的 X 和步骤 1 的 Y 矛盾）
   - 为什么冲突（具体的技术原因）
   - 建议方案（保留 1 调整 3 / 回到步骤 1 改方案 / 两个都改）
3. 只回看冲突的两个需求，**不影响已完成的步骤 2**
4. 用户选择后继续

### 验收超时

- 用户 2 分钟不回复 → 视为当前步骤通过，继续下一个
- 用户说"先跳过" → 记入待办，继续下一个

## UI/UX 特殊规则

### 架构先行

在任何视觉需求之前，必须确保以下基础设施就位：

1. **CSS 变量体系** — 颜色、间距、字体、圆角统一用变量
2. **公共组件库** — 按钮、输入框、面板、弹窗等基础元素从库中取
3. **布局系统** — 统一的 flex/grid 布局策略

后续所有 UI 元素只从组件库中取，**禁止组件内联样式**。

### 按空间区域拆 UX

以大区块为单位，和 VSCode 的结构方式一致：

```
顶栏          ← 父需求：整体结构
├── 文件菜单   ← 子需求：点击反馈（下拉/消失/层级）
│   └── 新建   ← 孙需求：具体动作
├── 编辑菜单
└── 帮助菜单

侧栏          ← 父需求
├── 工具列表   ← 子需求
└── 运行记录   ← 子需求

主区域        ← 父需求
├── 终端网格   ← 子需求
└── 空状态     ← 子需求
```

## 与 brainstorming 的选择

| 场景 | 用哪个 |
|------|--------|
| 功能边界模糊、需要探索 | incremental-iteration |
| 多个功能点、需要快速出 MVP | incremental-iteration |
| UI 密集型、需要反复调整 | incremental-iteration |
| 功能边界清晰、模块独立 | brainstorming |
| 需要完整设计文档才能开工 | brainstorming |
| 用户说「先设计再动手」 | brainstorming |
| 用户说「先跑起来再优化」 | incremental-iteration |

---

## AI 自检清单

每个阶段切换前，AI 必须确认：

- [ ] 收集阶段：我是否只记需求，没有动手？
- [ ] 审查阶段：我是否按 P0→P4 逐层扫过？
- [ ] 拆解阶段：我是否保持了用户的需求顺序？
- [ ] 执行阶段：我是否每个步骤都等了用户验收？
- [ ] 冲突发生时：我是否暂停了，而不是继续往前冲？
- [ ] UI 相关：基础设施是否先于视觉需求就位？
