# Incremental Iteration（增量迭代）

## 定位

与 superpowers 同级的工作流技能。不做一次性的完整设计，而是分阶段收需求、拆解、逐项交付。架构规则由 [[core-extension-philosophy]] 定义。

## 核心原则

1. **核心优先设计** — 核心接口和消息规范先于扩展就位，设计冻结后不轻改
2. **扩展隔离** — 扩展之间零依赖，只通过核心通信；相同逻辑抽离为公共工具库
3. **想到哪里做到哪里** — 上一步是下一步的基础，不耦合时可并行
4. **最小可验收单元** — 每步交付用户可感知的变化，有明确的 Definition of Done
5. **UI 架构先行** — 组件库和样式体系在视觉需求之前就位；按空间区域拆 UX
6. **技术债可见** — 不隐藏妥协和 hack，登记到技术债表，后续回顾清算

## 工作流总览

```
收集 → 审查 → 拆解(DoD+风险+分级) → 逐项执行 → Cool-down
  │                │                    │              │
  └── 三选项弹窗    └── 子需求递归       ├── 🔴验收     ├── 回顾技术债
                                        ├── 🟢静默     ├── 清理调试
                                        └── 冲突？暂停  └── 更新文档
```

## 阶段一：收集（只收需求，不动手）

1. 用户说需求 → AI 编号记录（不写代码、不设计架构、不给出实现方案）
2. 每轮输入后弹窗三个选项：**确认进入审查** / **继续补充** / **给我点建议**

**建议维度参考**（用户点「建议」时提示）：生命周期、状态管理、边界情况、用户体验、安全性

## 阶段二：审查（AI 补漏，不写代码）

按优先级逐层扫描，输出审查报告：

| 优先级 | 维度 | 如果漏了 | 检查点 |
|--------|------|---------|--------|
| **P0** | 安全 | 返工 | 输入校验、路径遍历、RCE、API key 泄露 |
| **P1** | 依赖 | 重来 | 需求 N 依赖需求 M 的产出？顺序合理？ |
| **P2** | 生命周期 | 用不了 | 启动/运行/退出/恢复/错误处理/持久化 |
| **P3** | UI/UX 边界 | 体验差 | 空状态、加载态、错误态、边界数据 |
| **P4** | 代码规范 | 可后修 | 行数≤300、核心/扩展边界、样式抽离 |

→ 弹窗：「确认，开始拆解」/「根据建议补充需求」

## 阶段三：拆解（保顺序 + DoD + 风险标记）

**规则**：按用户原顺序拆；无法保持时告知原因等确认。子需求边界模糊 → 回到收集阶段。

**拆解示例**：

```
需求: "加一个深色模式"
├── 步骤1: CSS变量定义 → DoD: tokens.css含主题色值  [🟢低  🟢静默]
├── 步骤2: 切换按钮     → DoD: 点击切换无报错      [🟡中  🔴验收]
└── 步骤3: 持久化        → DoD: 重启后主题保持      [🔴高  🔴验收]
```

**步骤分级**：

| 级别 | 判断标准 | 交互 |
|------|---------|------|
| 🟢 静默 | 不改变用户可感知行为（格式化、重命名、注释、提取内部函数） | 告知后继续 |
| 🔴 验收 | 改变可感知行为（新增交互、API变更、UI变化、数据变更） | 弹窗等确认 |

> 不能假设用户「没回复就是通过」，CLI 无法感知用户是否看到。

**风险标记**：🟢纯新增 → 🟡修改现有 → 🔴涉及持久化/公共接口/启动流程

→ 弹窗：「确认，开始执行」/「调整」

## 阶段四：逐项执行

1. 🟢 步骤：完成后告知 → 继续下一步
2. 🔴 步骤：完成后弹窗「通过，下一步」/「有问题」→ 等确认后继续
3. 两个模块不耦合 → 可并行（先声明依赖关系）

**技术债登记**：执行中遇到的妥协/hack/待重构项实时登记：

```
| ID | 来源步骤 | 描述 | 状态 |
| D1 | 步骤2 | 按钮缺动画 | ⏳ |
| D2 | 步骤5 | 配置读取重复 | ⏳ |
```

**冲突处理**：步骤 N 和已完成步骤 M 冲突 → 暂停 → 弹窗告知（冲突内容 + 原因 + 建议方案 + 影响面）→ 只回看 M 和 N，不影响其他。

## 阶段五：Cool-down（收尾）

| # | 检查项 |
|---|--------|
| 1 | 技术债回顾：是否将某些 debt 转为下一步需求 |
| 2 | 重复代码：跨步骤重复逻辑 → 提取到核心工具库 |
| 3 | 核心边界：扩展之间无直接引用 |
| 4 | 文件行数：新增文件是否超过 300 行 |
| 5 | 调试残留：console.log、临时 debug 文本 |
| 6 | 编译状态：tsc 双端零错误 |
| 7 | 文档更新：README/CHANGELOG/roadmap |
| 8 | 提交代码：Conventional Commits |

→ 弹窗：「结束本次迭代」/「处理技术债」

## 与 brainstorming 的选择

| 场景 | 技能 |
|------|------|
| 需求模糊、快速出 MVP、UI 密集、多迭代 | **incremental-iteration** |
| 功能边界清晰、需完整设计文档 | **brainstorming** |

## AI 自检清单

**收集**: 只记不写 / 三选项弹窗 / 需求已编号
**审查**: P0→P4 逐层 / 核心扩展边界 / 用户能懂
**拆解**: 保顺序 / 每步有 DoD / 🟢🔴 已区分 / 风险已标
**执行**: 🔴 等验收 / 妥协登记 / 冲突暂停
**Cool-down**: 技术债表 / 无调试残留 / 编译通过 / 已提交
