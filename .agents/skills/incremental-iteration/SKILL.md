# Incremental Iteration（增量迭代）

## 定位

与 superpowers 同级的工作流技能。用于需求模糊、需要快速出东西、UI 密集的场景。不做一次性的完整设计，而是分阶段收需求、拆解、逐项交付。

## 架构哲学

### 核心 vs 扩展

每个项目拆解为**核心**和**扩展**两部分：

| | 核心 | 扩展 |
|------|------|------|
| 角色 | 项目地基，不可变基线 | 可增删改的功能模块 |
| 稳定性 | 设计好之后不进行大范围改动 | 随时可迭代 |
| 依赖 | 不依赖任何扩展 | 只能依赖核心，不能依赖其他扩展 |
| 通信 | 定义接口和消息规范 | 通过核心提供的统一机制收发消息/事件 |
| 示例 | ConfigAdapter 接口、Registry、IPC 通道 | ClaudeCodeAdapter、Sidebar、TerminalView |

**规则**：
- 扩展之间零依赖。两个扩展有相同逻辑时 → 抽离到核心工具库
- 新增能力 = 扩展核心接口（加方法），不修改已有接口签名
- 判断放哪里：如果删除这个功能，核心还能不能跑？能 → 扩展；不能 → 核心

### 单文件约束

| 规则 | 理由 |
|------|------|
| 一个文件只做一件事 | 降低理解成本，AI 和人都能一眼看懂 |
| 单文件不超过 300 行 | 认知负载上限，超过强制拆分 |
| 拆分方式：`Component.tsx` → `Component/SubFeature.tsx` + `utils.ts` | 保持目录结构扁平可导航 |

---

## 核心原则

1. **核心优先设计** — 核心接口和消息规范先于所有扩展之前就位，设计冻结后不轻改
2. **扩展隔离** — 扩展之间零依赖，只通过核心通信。相同逻辑抽离为公共工具库
3. **想到哪里做到哪里** — 上一步是下一步的基础，两个模块不耦合时可以并行
4. **最小可验收单元** — 每次交付用户可感知的变化，每步有明确的 Definition of Done
5. **UI 架构先行** — 组件库和样式体系在视觉需求之前就位
6. **按空间区域拆 UX** — 以大区块为父需求，交互反馈为子，细节为孙
7. **技术债可见** — 不隐藏妥协和 hack，登记到技术债表，后续回顾清算

---

## 工作流总览

```
收集 ──→ 审查 ──→ 拆解（含DoD + 风险标记）──→ 逐项执行 ──→ Cool-down
  │        │         │                          │               │
  │        │         │                          ├── 🔴 需验收   │  回顾技术债
  │        │         │                          ├── 🟢 静默     │  清理调试代码
  │        │         │                          └── 冲突？暂停  │  更新文档
  │        │         │
  │        │         └── 子需求递归
  │        │
  │        └── 用户补充 → 回审查
  │
  └── 「给我点建议」→ AI 提示遗漏方向
```

---

## 第一阶段：收集（只收需求，不动手）

### AI 行为

1. 用户说出一段需求 → AI 即时记录到对话中（编号列表）
2. 引导用户继续说
3. **绝对禁止**：此阶段不写代码、不设计架构、不给出实现方案

### 交互控制（使用 AskUserQuestion 弹窗）

每轮用户输入后，AI 弹出三个选项：

| 选项 | 行为 |
|------|------|
| ✅ 确认，进入审查 | 结束收集，开始第二阶段 |
| 💬 继续补充 | 关闭弹窗，等待用户新一轮输入 |
| 💡 给我点建议 | AI 回顾已收集的需求，提示可能的遗漏方向（只提示维度，不补充具体内容） |

### 建议维度（用户点「给我点建议」时参考）

| 维度 | 提示方向 |
|------|---------|
| 生命周期 | 启动/初始化/运行中/退出/恢复/异常 |
| 状态管理 | 数据从哪来/存哪去/谁读写/什么触发变更 |
| 边界情况 | 空数据/超大值/网络断开/并发/权限不足 |
| 用户体验 | 首次使用/频繁操作/误操作/等待反馈 |
| 安全性 | 输入校验/路径安全/权限隔离/敏感信息 |

---

## 第二阶段：审查（AI 补漏，不写代码）

### AI 行为

按优先级逐层审查用户需求，输出审查报告：

```
## 审查报告

### P0 安全
- ✅ 无安全风险
- ⚠️ 路径由用户手动输入，未校验路径遍历 → 建议添加路径白名单校验
- ❌ API key 在配置中明文存储 → 建议使用 Electron safeStorage 加密

### P1 依赖关系
- ✅ 需求按 1→2→3 顺序无依赖冲突

### P2 生命周期
- ⚠️ 未定义应用退出时如何保存状态 → 建议在 onCleanup 中自动持久化
```

### 审查优先级

| 优先级 | 维度 | 如果漏了 | 检查点 |
|--------|------|---------|--------|
| **P0** | 安全 | 架构级返工 | 输入校验、路径遍历、RCE、API key 泄露、权限越界 |
| **P1** | 依赖关系 | 推倒重来 | 需求 N 是否依赖需求 M 的产出？顺序是否合理？ |
| **P2** | 生命周期 | 用不了 | 启动/运行/退出/恢复/错误处理/状态持久化 |
| **P3** | UI/UX 边界 | 体验差 | 空状态、加载态、错误态、边界数据、响应式 |
| **P4** | 代码规范 | 可后修 | 文件行数(≤300)、单文件单职责、命名一致性、样式抽离、类型覆盖、核心/扩展边界 |

**原则**：假设用户 0 编码经验，发现问题用用户能理解的语言解释。

### 审查后交互

输出审查报告 → 弹窗：「确认，开始拆解」/「根据建议补充需求」

---

## 第三阶段：拆解（保持用户顺序 + DoD + 风险标记）

### 规则

1. **默认按用户说的顺序**拆解。用户说 1,2,3,4 → 执行顺序就是 1→2→3→4
2. **无法按顺序时**（需求 2 依赖需求 4 的产物）→ 告诉用户为什么调整，给出调整后顺序，等确认
3. **子需求递归**：
   - 边界清晰 → 直接拆成执行步骤，不需要重新走全部流程
   - 边界模糊 → 回到收集阶段，仅针对该子需求补充

### 拆解粒度

每个最小单元以**用户可感知的功能变化**为界：

```
需求: "加一个深色模式"
├── 步骤 1: CSS 变量体系 + 色值定义
│   DoD: tokens.css 包含 dark 主题色值，不改变当前视觉
│   风险: 🟢 低（纯新增，不影响已有功能）
│   回滚: 删除变量文件即可
│
├── 步骤 2: 添加切换按钮
│   DoD: 点击按钮 → 主题切换 → 页面不刷新 → 控制台无报错
│   风险: 🟡 中（修改现有组件）
│   回滚: git revert
│
└── 步骤 3: 持久化到设置
    DoD: 切换主题 → 关闭应用 → 重新打开 → 主题保持
    风险: 🔴 高（涉及 Electron store 写入，可能影响已有设置）
    回滚: 清除 settings.json 中对应 key
```

### 步骤分级（决定确认方式）

| 级别 | 判断标准 | 交互方式 |
|------|---------|---------|
| 🟢 静默 | 不改变任何用户可感知行为（格式化、重命名、加注释、提取内部函数） | 完成后告知，直接继续下一步 |
| 🔴 需验收 | 改变用户可感知的行为（新增交互、API 变更、UI 变化、数据变更） | 完成后弹窗：「通过，下一步」/「有问题，描述一下」 |

> **注意**：不能假设用户"看了没回复就是通过"，因为 CLI 工具无法感知用户是否看到。静默仅适用于客观上不影响用户行为的变更。

### 风险标记

每个步骤必须标注：

| 标记 | 含义 | 处理 |
|------|------|------|
| 🟢 低 | 纯新增，不影响已有功能 | 正常执行 |
| 🟡 中 | 修改现有组件/接口，有回滚路径 | 额外验证相关功能 |
| 🔴 高 | 涉及数据持久化/公共接口/启动流程，回滚复杂 | 先备份相关代码，执行后强制用户验收 |

### 拆解后确认

输出完整拆解方案（含 DoD + 风险标记）→ 弹窗：「确认，开始执行」/「调整」

---

## 第四阶段：逐项执行

### 规则

1. 执行步骤 1 → 测试 → 🔴 步骤弹窗验收 → 通过后执行步骤 2
2. 🟢 步骤完成后告知用户，直接继续；🔴 步骤必须等用户确认
3. 两个完全不耦合的模块 → 可同时执行（先声明依赖关系给用户确认）

### 技术债登记

执行过程中遇到以下情况时，AI 主动登记到技术债表：

| 登记时机 | 格式 |
|---------|------|
| 需要但本次不做的优化 | `[DEBT] 按钮缺少过渡动画 → 待后续需求` |
| 已知的 hack/妥协 | `[DEBT] 颜色值硬编码 → 待迁移到 CSS 变量` |
| 应该重构但先跑通 | `[DEBT] 重复逻辑在 X.tsx 和 Y.tsx → 待提取公共函数` |

```
## 技术债登记表

| ID | 来源步骤 | 描述 | 影响 | 状态 |
|----|---------|------|------|------|
| D1 | 步骤 2 | 按钮缺少 hover 动画 | 视觉体验 | ⏳ 待处理 |
| D2 | 步骤 5 | 配置读取逻辑重复 | 维护性 | ⏳ 待处理 |
```

### 冲突处理

执行步骤 N 时，发现和已完成的步骤 M 冲突：

1. **立即暂停**全部执行
2. 弹窗告知用户：
   - 冲突内容（步骤 N 的 X 和步骤 M 的 Y 矛盾）
   - 为什么冲突（技术原因）
   - 建议方案（保留 M 调整 N / 回到 M 改方案 / 两个都改）
   - 影响面（是否影响其他已完成的步骤）
3. 只回看冲突的两个需求，**不影响其他已完成步骤**
4. 用户选择后继续

### 验收弹窗格式

🔴 步骤完成后，弹窗：

```
步骤 2/5「添加切换按钮」已完成

验收标准:
  ✅ 点击按钮 → 主题切换
  ✅ 页面不刷新
  ✅ 控制台无报错

请验收后选择：
```

选项：「通过，下一步」/「有问题，描述一下」

---

## 第五阶段：Cool-down（收尾清理）

全部步骤执行完后，执行 Cool-down 检查：

### Cool-down 清单

| # | 检查项 | 说明 |
|---|--------|------|
| 1 | **技术债回顾** | 展示登记表，询问用户是否将某些 debt 转为下一步需求 |
| 2 | **重复代码** | 跨步骤有无重复逻辑 → 提取到核心工具库 |
| 3 | **核心边界** | 确认无扩展越界（扩展之间不直接引用） |
| 4 | **文件行数** | 新增文件是否超过 300 行 → 即时拆分 |
| 5 | **调试残留** | 检查 console.log、临时 debug 文本框、临时代码 |
| 6 | **编译状态** | `tsc --noEmit` 无错误，`tsc -p main/tsconfig.json` 无错误 |
| 7 | **文档更新** | 是否需要更新 README、CHANGELOG、roadmap |
| 8 | **提交代码** | 编写 Conventional Commits 格式的 commit message |

### 输出示例

```
## Cool-down 报告

技术债登记表:
  D1: 按钮缺少 hover 动画 → 建议下次需求处理
  D2: 配置读取逻辑重复 → 已在下个迭代中计划

清理检查:
  ✅ 已清除 2 处 console.log
  ✅ tsc 编译零错误
  ✅ README 已更新
  ✅ 已提交: feat(settings): 深色模式切换

下一步建议:
  1. 处理技术债 D1（按钮动画）
  2. 或者开始新需求「壁纸管理」
```

### Cool-down 后交互

输出报告 → 弹窗：「结束本次迭代」/「处理技术债」

## UI/UX 特殊规则

### 架构先行

在任何视觉需求之前，必须确保以下基础设施就位：

1. **CSS 变量体系** — 颜色、间距、字体、圆角统一用变量
2. **公共组件库** — 按钮、输入框、面板、弹窗等基础元素从库中取
3. **布局系统** — 统一的 flex/grid 布局策略

后续所有 UI 元素只从组件库中取，**禁止组件内联样式**。

### 按空间区域拆 UX

以大区块为单位，和 VSCode 的结构方式一致：

```
顶栏          ← 父需求：整体结构
├── 文件菜单   ← 子需求：点击反馈（下拉/消失/层级）
│   └── 新建   ← 孙需求：具体动作
├── 编辑菜单
└── 帮助菜单

侧栏          ← 父需求
├── 工具列表   ← 子需求
└── 运行记录   ← 子需求

主区域        ← 父需求
├── 终端网格   ← 子需求
└── 空状态     ← 子需求
```

## 与 brainstorming 的选择

| 场景 | 用哪个 |
|------|--------|
| 功能边界模糊、需要探索 | incremental-iteration |
| 多个功能点、需要快速出 MVP | incremental-iteration |
| UI 密集型、需要反复调整 | incremental-iteration |
| 功能边界清晰、模块独立 | brainstorming |
| 需要完整设计文档才能开工 | brainstorming |
| 用户说「先设计再动手」 | brainstorming |
| 用户说「先跑起来再优化」 | incremental-iteration |

## AI 自检清单

每个阶段切换前，AI 必须确认：

### 收集阶段
- [ ] 我是否只记需求，没有动手写代码？
- [ ] 我是否给用户提供了三个选项弹窗？
- [ ] 需求是否已编号并记录在对话中？

### 审查阶段
- [ ] 我是否按 P0→P4 逐层扫过？
- [ ] 我是否检查了新增代码是否放对了核心/扩展位置？
- [ ] 我是否检查了扩展之间有无不当耦合？
- [ ] 我是否用用户能理解的语言解释问题？
- [ ] 我是否给用户提供了「确认拆解」vs「补充需求」的选择？

### 拆解阶段
- [ ] 我是否保持了用户的需求顺序？
- [ ] 每个步骤是否都有 Definition of Done？
- [ ] 每个步骤是否都标注了工作量和风险等级？
- [ ] 我是否区分了 🟢 静默步骤和 🔴 需验收步骤？

### 执行阶段
- [ ] 每个 🔴 步骤是否都弹窗等待了用户验收？
- [ ] 遇到妥协/hack 时是否登记到技术债表？
- [ ] 冲突发生时是否立即暂停并告知用户？
- [ ] 🟢 步骤是否完成后告知用户？

### Cool-down 阶段
- [ ] 是否展示了技术债登记表？
- [ ] 是否检查并清理了调试残留？
- [ ] 是否确认了编译通过？
- [ ] 是否建议了下一步行动？

### UI + 架构 相关
- [ ] 基础设施（CSS 变量/组件库/布局）是否先于视觉需求就位？
- [ ] 是否有组件内联了 style？如有，是否登记为技术债？
- [ ] 新增文件的职责是否放在正确的核心/扩展位置？
- [ ] 扩展之间是否存在直接引用（import 另一个扩展的内部模块）？
- [ ] 新增文件是否超过 300 行？是否需要立即拆分？
